# Student Minimum-Pass Solution — Testing Guide

**Purpose:** Step-by-step validation of `student_CapstoneTools.psm1` and `student_Invoke-Capstone.ps1` against the CVTE range environment.

**Intent:** Confirm that the minimum-pass student solution satisfies every rubric item (R1–R4) under realistic conditions. If it fails here, the rubric wording needs tightening.

**Scope:** Instructor-only. This is a calibration exercise, not a student-facing workflow.

---

## 📋 Pre-Flight Checklist

Before starting, confirm each:

- [ ] Ops workstation accessible, logged in as `student`
- [ ] PowerShell 7 installed on Ops
- [ ] `System-1`, `System-2`, `System-3` reachable and powered on
- [ ] `student` account on each target has completed any first-logon password reset
- [ ] Same password set on Ops\student, System-1\student, System-2\student, System-3\student
- [ ] TrustedHosts on Ops includes all three targets
- [ ] WinRM running on Ops and all targets
- [ ] Sysmon staged on Ops: `C:\Tools\Sysmon\Sysmon64.exe` and `C:\Tools\Sysmon\sysmonconfig-export.xml`
- [ ] `student_CapstoneTools.psm1` and `student_Invoke-Capstone.ps1` copied to `C:\Users\Student\` on Ops

## Phase 0 — Environment Verification

### 0.1 Confirm you're on Ops with PowerShell 7

```powershell
hostname
$PSVersionTable.PSVersion
```

**Pass:** hostname is Ops, PSVersion is 7.x.

### 0.2 Confirm student files are present

```powershell
cd C:\Users\Student
Get-ChildItem student_CapstoneTools.psm1, student_Invoke-Capstone.ps1 | Select-Object Name, Length, LastWriteTime
```

**Pass:** both files listed, non-zero length.

### 0.3 Confirm WinRM, TrustedHosts, target reachability

```powershell
Get-Service WinRM
(Get-Item WSMan:\localhost\Client\TrustedHosts).Value
'System-1','System-2','System-3' | ForEach-Object { try { Test-WSMan -ComputerName $_ -ErrorAction Stop | Out-Null; "$_ : OK" } catch { "$_ : FAIL" } }
```

**Pass:** WinRM Running, TrustedHosts lists all three systems, all three targets OK.

### 0.4 Build test credential

```powershell
$password = ConvertTo-SecureString 'student' -AsPlainText -Force
$cred = New-Object System.Management.Automation.PSCredential('.\student', $password)
$cred.UserName
```

**Pass:** `$cred.UserName` returns `.\student`.

> **Note:** Using the `.\` portable scope so one credential works against all three targets. If this form doesn't work in your environment, substitute per-target scoping (`System-1\student`, etc.) and rebuild the credential between phases.

### 0.5 Auth smoke test

```powershell
$s = New-PSSession -ComputerName System-1 -Credential $cred -Authentication Negotiate
$s
Remove-PSSession $s
```

**Pass:** session opens cleanly, State `Opened`, then removes without error.

**If this fails:** stop. Fix auth before proceeding. Solutions won't test meaningfully against broken auth.

## Phase 1 — Module Function Standalone Tests (R1, R3)

### 1.1 Import student module

```powershell
Import-Module C:\Users\Student\student_CapstoneTools.psm1 -Force
Get-Command -Module student_CapstoneTools
```

**Pass:** three functions listed — `Get-RemoteProcessInfo`, `Get-RemoteServiceInfo`, `Install-SysmonRemote`.

### 1.2 Test Get-RemoteProcessInfo (R1 partial)

```powershell
$procs = Get-RemoteProcessInfo -ComputerName System-1 -Credential $cred
$procs | Measure-Object | Select-Object Count
$procs | Select-Object -First 2 | Format-List
```

**Pass criteria:**
- Count > 0 (any value ~100+ is reasonable for a typical Windows host)
- Each object is a `PSCustomObject` with `ComputerName`, `Name`, `ProcessId`, `Path` fields
- `ComputerName` field populated with the target name

**This validates R1 for the first Get-* function.**

### 1.3 Test Get-RemoteServiceInfo (R1 complete)

```powershell
$svcs = Get-RemoteServiceInfo -ComputerName System-1 -Credential $cred
$svcs | Measure-Object | Select-Object Count
$svcs | Select-Object -First 2 | Format-List
```

**Pass criteria:**
- Count > 0 (~200-300 services typical)
- Each object is a `PSCustomObject` with `ComputerName`, `Name`, `State`, `StartMode` fields

**R1 now fully validated:** two Get-* functions, both query remote systems, both return PSCustomObject, both retrieve data from ≥ 2 distinct hosts (we'll confirm host 2 in Phase 2).

### 1.4 Test Install-SysmonRemote (R3)

```powershell
$result = Install-SysmonRemote -ComputerName System-1 -Credential $cred -BinaryPath 'C:\Tools\Sysmon\Sysmon64.exe' -ConfigPath 'C:\Tools\Sysmon\sysmonconfig-export.xml'
$result
```

**Pass criteria:**
- Returns a `PSCustomObject` with `ComputerName` and `SysmonState` fields
- `SysmonState` equals `Running`

### 1.5 Independently verify Sysmon is running on System-1

```powershell
Invoke-Command -ComputerName System-1 -Credential $cred -Authentication Negotiate -ScriptBlock { Get-Service Sysmon64 }
```

**Pass:** Service `Sysmon64` with Status `Running`.

**R3 validated.** Module standalone tests complete.

## Phase 2 — Orchestrator Multi-Target Test (R2 initial)

### 2.1 Clean up sessions before orchestrator test

```powershell
Get-CimSession | Remove-CimSession
Get-PSSession | Remove-PSSession
```

### 2.2 Run orchestrator against multiple targets, no deploy

```powershell
$modPath = (Resolve-Path C:\Users\Student\student_CapstoneTools.psm1).Path
.\student_Invoke-Capstone.ps1 -ComputerName 'System-1','System-2','System-3' -Credential $cred -ModulePath $modPath
```

**Pass criteria:**
- Script completes without unhandled errors
- Final summary: `3 succeeded, 0 failed`
- Each target shows `ProcessCount > 0` and `ServiceCount > 0`
- Each target has `Status: Success` and `Error: null`

**R1 now fully validated** — data retrieved from ≥ 2 distinct remote hosts (three, in fact).

### 2.3 Confirm execution was actually concurrent (R2)

Re-run with a single-target throttle to force serial execution, then compare:

```powershell
$serialTime = Measure-Command { .\student_Invoke-Capstone.ps1 -ComputerName 'System-1','System-2','System-3' -Credential $cred -ModulePath $modPath }
$serialTime.TotalSeconds
```

Note: the student orchestrator hardcodes `-ThrottleLimit 5`, so to test serial vs parallel you'd need to temporarily edit the script. For grading validation, concurrency can be inferred from total run time: if 3 targets complete faster than 3x the per-target time from Phase 1, execution was parallel.

**Alternative concurrency proof:** watch Ops's Task Manager during the run — multiple `pwsh.exe` worker processes appear under the parent PowerShell during parallel dispatch.

**R2 validated** — `ForEach-Object -Parallel` is running jobs concurrently.

## Phase 3 — Orchestrator With Deploy (R3 multi-target)

### 3.1 Uninstall Sysmon first (optional — for clean deploy test)

If Sysmon was installed during Phase 1.4, you may want to uninstall before the multi-target deploy to confirm fresh installs work:

```powershell
'System-1' | ForEach-Object {
    Invoke-Command -ComputerName $_ -Credential $cred -Authentication Negotiate -ScriptBlock {
        & 'C:\Windows\Temp\Sysmon64.exe' -u force | Out-Null
    }
}
```

Optional. Skipping this means Phase 3 will re-install Sysmon on System-1 (which works, just less clean a signal).

### 3.2 Run orchestrator with deploy enabled

```powershell
.\student_Invoke-Capstone.ps1 -ComputerName 'System-1','System-2','System-3' -Credential $cred -ModulePath $modPath -DeploySysmon -SysmonBinary 'C:\Tools\Sysmon\Sysmon64.exe' -SysmonConfig 'C:\Tools\Sysmon\sysmonconfig-export.xml'
```

**Pass criteria:**
- Final summary: `3 succeeded, 0 failed`
- Each target shows `SysmonState: Running`

### 3.3 Independently verify Sysmon is running on all three targets

```powershell
'System-1','System-2','System-3' | ForEach-Object {
    $result = Invoke-Command -ComputerName $_ -Credential $cred -Authentication Negotiate -ScriptBlock { (Get-Service Sysmon64 -ErrorAction SilentlyContinue).Status }
    "$_ : $result"
}
```

**Pass:** all three show `Running`.

**R3 validated at multi-target scale.**

## Phase 4 — Error Handling Test (R4)

### 4.1 Induce a per-target failure

Include a non-existent hostname in the target list:

```powershell
.\student_Invoke-Capstone.ps1 -ComputerName 'System-1','NonExistent-99','System-3' -Credential $cred -ModulePath $modPath
```

**Pass criteria:**
- Script completes without crashing
- Final summary: `2 succeeded, 1 failed`
- `System-1` and `System-3` show `Status: Success`
- `NonExistent-99` shows `Status: Failed` with an `Error` message populated
- No script-level exception propagated to the console

**R4 validated** — errors handled per-target, successful targets distinguished from failed.

### 4.2 Induce a credential failure

Test with a deliberately wrong credential:

```powershell
$badPass = ConvertTo-SecureString 'wrongpassword' -AsPlainText -Force
$badCred = New-Object System.Management.Automation.PSCredential('.\student', $badPass)
.\student_Invoke-Capstone.ps1 -ComputerName 'System-1','System-2' -Credential $badCred -ModulePath $modPath
```

**Pass criteria:**
- Script completes
- Both targets show `Status: Failed` with Access Denied or similar error
- No crash, no hang

**R4 validated under auth failure scenario.**

## Phase 5 — Known Weakness Demonstrations (Instructor-Only)

These steps demonstrate the gaps in the student solution vs. the reference. They are **not** rubric failures, but are useful for calibration and class discussion.

### 5.1 Session leak test

The student module only removes sessions in the success path. Induce an error and check for leaked sessions:

```powershell
Get-CimSession
Get-PSSession
```

After Phase 4.2 (bad credential), these should both return empty — but note the student code does not have `finally` blocks, so if the session creation itself succeeded but a later call failed, the session would be orphaned. This is a theoretical concern for the student code; hard to trigger cleanly in practice.

**Calibration note:** Not a rubric failure. Reference has `finally` block; student does not. Both satisfy R4 as written.

### 5.2 Install exit-code gap

The student `Install-SysmonRemote` ignores the installer exit code. Demonstrate:

```powershell
# Pass a bad config path to the installer. The student function will still report "Running" if Sysmon was previously installed.
Install-SysmonRemote -ComputerName System-1 -Credential $cred -BinaryPath 'C:\Tools\Sysmon\Sysmon64.exe' -ConfigPath 'C:\Tools\Sysmon\doesnotexist.xml'
```

**Expected behavior:** `Copy-Item` fails because the config doesn't exist. The `try/catch` catches this and returns `SysmonState: Failed`. Good — the function handles this case.

**Calibration note:** The edge case that would bypass detection is a successful file copy followed by an installer failure that leaves the service in a Running state from a prior install. Hard to induce in testing, but worth noting.

## 📊 Rubric Validation Summary

After completing all phases, you should be able to confirm:

| Rubric | Phase(s) | Pass Evidence |
|---|---|---|
| R1 — Module has 2+ Get-* functions querying ≥ 2 remote systems | 1.2, 1.3, 2.2 | Both functions return PSCustomObject data from System-1 standalone, and from all three systems via orchestrator |
| R2 — Orchestrator executes in parallel | 2.2, 2.3 | `ForEach-Object -Parallel` runs concurrently (verifiable by timing or process inspection) |
| R3 — Deploy function installs Sysmon remotely | 1.4, 1.5, 3.2, 3.3 | Sysmon service Running on all three targets after orchestrator run |
| R4 — Script handles errors without crashing | 4.1, 4.2 | Offline target and bad credential both handled per-target; script completes; successful vs. failed distinguished in output |

If all four rubric items have pass evidence, the student solution clears the minimum bar.

---

## ⚠️ What To Do If A Phase Fails

**If Phase 0 fails:** environment issue, not a solution issue. Fix environment first.

**If Phase 1 fails:** the student module has a bug that should fail rubric R1 or R3. This is a genuine grading signal — document what failed, then decide if the rubric as written catches this.

**If Phase 2 fails on concurrency:** R2 may need tighter wording. "Demonstrably concurrent" leaves some room for interpretation.

**If Phase 3 fails with Sysmon not Running:** either R3 grading needs to be more forgiving (accept "installer exit code 0") or the student solution genuinely didn't install Sysmon. Investigate before declaring pass/fail.

**If Phase 4 fails (script crashes):** R4 is not satisfied. This is a clear fail.

**Calibration note:** The purpose of this test is to see whether the rubric wording catches what it should catch and accepts what it should accept. If any phase's outcome feels "wrong" relative to what you'd want to grade, the feedback goes back into the rubric — not into the student solution.
